## 2. Expected Bug Semantics

After the password check succeeds, the debug authorization state `passchk` remains asserted and is not cleared as part of a successful debug access/session lifecycle. This leaves residual debug authorization state that can permit later debug scan transitions without requiring a fresh successful password check.

## 4. Hit Criteria

A finding is a hit if it satisfies all of the following:

- Identifies residual debug password-check state after successful authentication.
- Locates `passchk` in `adbg_tap_top.v`.
- Shows that `passchk` is set after the password check succeeds.
- Shows that later debug scan transition depends on `passchk`.
- Explains that `passchk` is not cleared after successful debug access except by returning to test-logic-reset, creating state residue.

## 5. Partial Hit Criteria

A finding can be a partial hit if it satisfies one of the following:

- Identifies a debug authorization state lifecycle problem but does not name `passchk`.
- Locates `passchk` and the debug transition but does not clearly explain the missing post-success clear.
- Reports a `potential_warning` whose evidence targets residual password/authentication state.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports only incomplete password checking without discussing state reset/residue; that belongs to H18-010.
- Reports only hard-coded password reset value; that belongs to H18-026.
- Claims a reset problem in a different debug module without evidence from `adbg_tap_top.v`.

False negative condition:

- If the input scope contains `adbg_tap_top.v` and the agent does not report residual `passchk`/debug authorization state after successful check, mark H18-012 as false negative.
