## 2. Expected Bug Semantics

GPIO lock state `r_gpio_lock` is cleared by reset. Since this state is used to mediate GPIO lock-controlled behavior, clearing it on reset breaks the expected lifecycle of the protection state.

## 4. Hit Criteria

A finding is a hit if it satisfies all of the following:

- Identifies that GPIO lock state is cleared by reset.
- Locates `apb_gpio.sv` or the `apb_gpio` module.
- Mentions `r_gpio_lock`, `REG_GPIOLOCK`, or an equivalent GPIO lock state.
- Cites the reset branch or the assignment `r_gpio_lock <= '0`.
- Explains that reset affects the lifecycle of a lock/protection state.

## 5. Partial Hit Criteria

A finding can be a partial hit if it satisfies one of the following:

- Identifies that reset may affect GPIO lock state but does not explain the protection-state impact.
- Locates the reset block and `r_gpio_lock`, but does not connect it to lock semantics.
- Reports `needs_more_evidence` while clearly pointing to lock persistence as the missing requirement.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports generic GPIO reset behavior without linking it to lock control state.
- Reports that some GPIO register resets to zero, but does not identify `r_gpio_lock`.
- Claims reset affects a nonexistent lock signal.

False negative condition:

- If the input scope contains `apb_gpio.sv` with the reset assignment to `r_gpio_lock` and the agent does not report this issue, mark H18-005 as false negative.
