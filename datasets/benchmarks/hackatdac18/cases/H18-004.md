## 2. Expected Bug Semantics

GPIO lock register `r_gpio_lock` is software-writable through the APB write path. Since this register is itself used to mediate GPIO lock-controlled behavior, direct software writes to `REG_GPIOLOCK` weaken the lock/write-protection mechanism.

## 4. Hit Criteria

A finding is a hit if it satisfies all of the following:

- Identifies that the GPIO lock register or lock control state is software-writable.
- Locates `apb_gpio.sv` or the `apb_gpio` module.
- Mentions `REG_GPIOLOCK`, `r_gpio_lock`, or an equivalent GPIO lock state.
- Cites the APB write path or the assignment `r_gpio_lock <= PWDATA`.
- Explains that the lock register itself is not protected from software writes.

## 5. Partial Hit Criteria

A finding can be a partial hit if it satisfies one of the following:

- Identifies a GPIO lock or GPIO write-protection risk but does not show that `REG_GPIOLOCK` itself is writable.
- Locates `apb_gpio` and `r_gpio_lock`, but only explains how `r_gpio_lock` protects other registers.
- Reports a `potential_warning` whose semantics clearly relate to GPIO lock/write-protection.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports a generic GPIO register write risk without linking it to lock semantics.
- Reports `REG_PADDIR` or `REG_PADOUT` writability but does not mention `REG_GPIOLOCK` or `r_gpio_lock`.
- Claims a nonexistent GPIO lock signal or nonexistent guard.

False negative condition:

- If the input scope contains `apb_gpio.sv` with the `REG_GPIOLOCK` write branch and the agent does not report this issue, mark H18-004 as false negative.
