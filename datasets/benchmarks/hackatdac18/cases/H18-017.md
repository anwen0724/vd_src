## 2. Expected Bug Semantics

The design exposes core debug/register-file access through a memory-mapped APB debug path. The APB debug bridge drives debug requests into the core debug unit, and the debug unit can write general-purpose registers when the core is halted. The debug unit also exposes a debug jump path, so writable debug/register-file state can influence execution state and support code-injection style behavior.

## 4. Hit Criteria

A finding is a hit if it satisfies all of the following:

- Identifies memory-mapped debug/register-file access as the relevant vulnerability.
- Locates the APB debug bridge or core debug unit path.
- Mentions writable general-purpose register access through `regfile_wreq_o`, `debug_addr_i`, or equivalent.
- Explains how memory-mapped debug/register access can alter execution state or support code-injection behavior.

## 5. Partial Hit Criteria

- Finds memory-mapped debug access but not register-file write evidence.
- Finds register-file debug writes but not the APB bridge path.
- Reports a debug path code-execution risk with correct file references but incomplete signal tracing.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports generic JTAG exposure without memory-mapped register-file write evidence.
- Confuses this case with H18-009/H18-010/H18-012 password-check bugs.
- Claims arbitrary memory writes without showing the debug/register-file path.

False negative condition:

- If the input scope contains `fc_subsystem.sv` and `zeroriscy_debug_unit.sv`, and the agent does not report memory-mapped debug/register-file write access, mark H18-017 as false negative.

## 7. Evaluation Notes

- The key scoring target is not “debug exists”, but that APB-visible debug access reaches register-file write/read control and can affect execution state.
