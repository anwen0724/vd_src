## 2. Expected Bug Semantics

The advanced debug TAP contains password-checking logic, but the check is implemented as a cumulative per-bit match counter rather than a complete and well-scoped authentication comparison. The debug instruction path is gated by `passchk`; incorrect handling of `correct`, `bitindex`, and `passchk` weakens the debug authorization guard.

## 4. Hit Criteria

A finding is a hit if it satisfies all of the following:

- Identifies incorrect debug password/authentication checking logic.
- Locates `adbg_tap_top.v` or the advanced debug TAP logic.
- Mentions `passchk`, `correct`, `bitindex`, `pass`, or equivalent password-check state.
- Cites the cumulative bit comparison / counter logic around lines 249-258.
- Explains that this logic weakens or incorrectly implements debug authorization.

## 5. Partial Hit Criteria

A finding can be a partial hit if it satisfies one of the following:

- Identifies a debug authorization/password-check risk but does not locate `adbg_tap_top.v`.
- Locates the debug TAP state machine but does not explain why the password check is incorrect.
- Reports a `potential_warning` whose evidence points to `passchk`, `correct`, or `bitindex`.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports a generic debug/JTAG exposure without referencing the password-check logic.
- Claims a nonexistent password register or nonexistent authentication module.
- Confuses this case with the separate unprotected JTAG/LINT path in H18-028.

False negative condition:

- If the input scope contains `adbg_tap_top.v` and `adv_dbg_if.sv`, and the agent does not report the incorrect debug password-checking logic, mark H18-009 as false negative.
