## 2. Expected Bug Semantics

The official bug semantics are that the advanced debug unit does not enforce a complete 32-bit password check. In RTL, the relevant evidence is the password-check state machine in `adbg_tap_top.v`, where the 32-bit `pass` value is consumed through `pass[bitindex]` and authorization is granted through a counter threshold rather than a complete 32-bit equality check.

## 4. Hit Criteria

A finding is a hit if it satisfies all of the following:

- Identifies an incomplete or partial debug password check.
- Locates the password-checking logic in `adbg_tap_top.v`.
- Mentions the 32-bit `pass`, `bitindex`, `correct`, or the `correct >= 32'h0001_FFFF` threshold.
- Explains that access is granted without a complete 32-bit password equality check.
- Connects the incomplete check to debug authorization through `passchk` or debug scan access.

## 5. Partial Hit Criteria

A finding can be a partial hit if it satisfies one of the following:

- Identifies weak debug password checking but does not distinguish this partial-check case from H18-009.
- Locates `pass`, `bitindex`, or `correct` but does not explain the incomplete check semantics.
- Reports a `needs_more_evidence` warning that correctly targets password bit coverage or threshold-based authorization.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports only that the password is hard-coded; that belongs to H18-026, not this case.
- Reports only that password state is not reset after success; that belongs to H18-012.
- Claims a specific missing bit position without evidence from the visible RTL.

False negative condition:

- If the input scope contains `adbg_tap_top.v` and the agent does not report incomplete/partial debug password checking, mark H18-010 as false negative.
