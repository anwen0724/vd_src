## 2. Expected Bug Semantics

AES0 masks `key_big0` and `key_big1` when `debug_mode_i` is asserted, but `key_big2` is not masked. Key2 can therefore remain connected to the selected AES key path in debug mode.

| Key2 write map | `third_party/hackatdac21/piton/design/chip/tile/ariane/src/aes0/aes0_wrapper.sv` | 141-151 | `key2`, `reglk_ctrl_i[5]` | Shows key2 is configurable key state. |
| Core key selection | `third_party/hackatdac21/piton/design/chip/tile/ariane/src/aes0/aes0_wrapper.sv` | 197-206 | `key_big`, `key_big2`, `key_sel` | Shows key2 can feed the AES core key input. |

## 4. Hit Criteria

- Reports AES0 key2 is not cleared/masked in debug mode.
- Cites the inconsistent debug masking between `key_big0`, `key_big1`, and `key_big2`.
- Explains impact on AES key exposure/residue under debug.

## 5. Partial Hit Criteria

- Reports generic AES debug key issue but does not identify key2.
- Finds debug masking but misses the unmasked key path.

## 6. False Positive / False Negative Rules

False positive examples:

- Claims all AES keys are unmasked without evidence.
- Reports AES plaintext residue only.

False negative condition:

- If the input scope contains `aes0_wrapper.sv`, but the agent does not report unmasked AES key2 in debug mode, mark H21-047 as false negative.

## 7. Evaluation Notes

This case is debug/security-state handling, not a general AES algorithm issue.
