## 2. Expected Bug Semantics

The PKT wrapper receives fuse data through `fuse_rdata_i`. During reads, `rdata` is first assigned `fuse_rdata_i` before the address case statement. The default branch only conditionally overwrites `rdata`, so incomplete address/default handling can leave fuse data exposed through the read response.

## 4. Hit Criteria

- Reports PKT fuse data exposure through faulty default/read handling.
- Cites the `rdata = fuse_rdata_i` pre-assignment and conditional default handling.
- Explains why this can leak fuse data for unmapped or insufficiently covered address cases.

## 5. Partial Hit Criteria

- Reports generic PKT/fuse exposure without identifying the read default logic.
- Finds `fuse_rdata_i` but not its propagation to `rdata`.

## 6. False Positive / False Negative Rules

False positive examples:

- Reports PKT access-control coupling without evidence of fuse-data read leakage.
- Claims fuse memory contents are wrong rather than exposed.

False negative condition:

- If the input scope contains `pkt_wrapper.sv`, but the agent does not report the fuse data leakage through read/default handling, mark H21-059 as false negative.

## 7. Evaluation Notes

This is permission/security relevant because fuse data is a protected resource exposed through a peripheral interface.
