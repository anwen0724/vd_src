## 2. Expected Bug Semantics

The AES key is stored as a fixed ROM2 constant. ROM2 copies the constant into `secure_reg`, the peripheral harness routes `key_reg_out[0]` into the AES wrapper as `key_in`, and the AES wrapper assigns `key_big = key_in`. This makes the AES key a hardcoded RTL-visible secret.

## 3. Expected RTL Evidence

| Evidence | 文件 | 行号 | 关键对象 | 评分用途 |
|---|---|---:|---|---|
| Hardcoded AES key constant | `third_party/hackatdac19/src/rom2/rom2.sv` | 17-22 | `mem[0]` AES comment | Shows the AES key is stored as a fixed constant. |
| ROM2 constants copied to secure registers | `third_party/hackatdac19/src/rom2/rom2.sv` | 25-34 | `secure_reg`, `mem` | Shows constants become runtime key/configuration registers. |
| AES key routed from ROM2 output | `third_party/hackatdac19/tb/ariane_peripherals.sv` | 181-217, 463-468 | `key_reg_out[0]`, `key_in` | Shows ROM2 output is passed to the AES wrapper. |
| AES wrapper uses key input directly | `third_party/hackatdac19/src/aes/aes_wrapper.sv` | 32-57, 156-161 | `key_in`, `key_big`, `key` | Shows the hardcoded-derived key is used by AES. |

## 4. Hit Criteria

- Identifies that the AES key is hardcoded in ROM2 constants.
- Cites the ROM2 constant and the path into AES `key_in` / `key_big`.
- Explains why this is a hardcoded-secret issue.

## 5. Partial Hit Criteria

- Finds ROM2 key constants but does not connect them to AES.
- Reports AES key readback only; that is H19-010.

## 6. False Positive / False Negative Rules

False positive examples:

- Claims JTAG key hardcoding; that is H19-054.
- Claims AES oracle exposure without hardcoded key evidence; that is H19-014.

False negative condition:

- If the input scope contains `rom2.sv`, `ariane_peripherals.sv`, and `aes_wrapper.sv`, but the agent does not report the hardcoded AES key path, mark H19-015 as false negative.

## 7. Evaluation Notes

This case is about secret origin, not register readback.
