## 2. Expected Bug Semantics

After JTAG password success, `pass_chk` drives `umode_o` high in `dmi_jtag.sv`. The test harness wires this signal into the Ariane core as `umode_i`. Inside the CSR register file, `umode_i` is used in the privilege output expression so that `priv_lvl_o` becomes machine mode when `umode_i` is asserted. Thus a JTAG unlock signal can force the processor privilege output to machine mode.

## 4. Hit Criteria

- Identifies that unlocking JTAG can force machine privilege.
- Cites both the JTAG `pass_chk` to `umode_o` path and the CSR `priv_lvl_o` expression.
- Explains this is a privilege-control failure, not only a debug-access issue.

## 5. Partial Hit Criteria

- Finds the `umode_i` privilege expression but not the JTAG source.
- Reports a generic JTAG unlock problem without explaining machine-mode privilege escalation.

## 6. False Positive / False Negative Rules

False positive examples:

- Claims only that debug is enabled; that is H19-005.
- Claims only a missing JTAG password guard; that is H19-047.

False negative condition:

- If the input scope contains `dmi_jtag.sv`, `ariane_testharness.sv`, and `csr_regfile.sv`, but the agent does not report the JTAG-unlock-to-machine-privilege path, mark H19-048 as false negative.

## 7. Evaluation Notes

This case should be scored on the cross-module signal path from JTAG authentication state into processor privilege state.
